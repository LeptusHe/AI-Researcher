Example #1:
1. Title: Analyzing the Impact of Tile Size on Variable Rate Shading Effectiveness Across Mobile GPU Architectures

2. Problem Statement: This is an analysis paper. We analyze how different tile sizes (8x8, 16x16, 32x32) affect the effectiveness of Variable Rate Shading (VRS) on different mobile GPU architectures (Adreno, Mali, Apple GPU). This is interesting because mobile GPUs have different tile sizes in their tile-based rendering pipelines, and the optimal VRS tile granularity may vary across architectures.

3. Motivation: Existing VRS research focuses on desktop GPUs where tile size is relatively uniform. Mobile GPUs have diverse tile sizes (Adreno uses 32x32, Mali uses 16x16, Apple GPUs vary), and the interaction between VRS tile granularity and the native tile size of the GPU architecture has not been studied. Understanding this interaction is crucial for implementing effective VRS on mobile.

4. Proposed Method: We implement VRS with configurable tile sizes on three mobile GPU families. For each combination of VRS tile size and GPU architecture, we measure: (a) the overhead of the shading rate assignment pass, (b) the rendering quality at each coarse shading level (PSNR/SSIM vs full-rate reference), (c) the power savings achieved, (d) the bandwidth reduction. We also analyze whether aligning VRS tiles with the GPU's native tile size provides benefits.

5. Step-by-Step Experiment Plan:
   Step 1: Implement a configurable VRS system in Vulkan that supports 8x8, 16x16, and 32x32 tile sizes.
   Step 2: Prepare 5 diverse test scenes: Sponza (indoor, moderate detail), Bistro (mixed, high detail), San Miguel (outdoor, vegetation-heavy), a simple terrain scene, and a dense urban scene.
   Step 3: On each device (Snapdragon 8 Gen 2, Dimensity 9200, Apple A17 Pro), render each scene with each VRS tile size at three shading rate configurations: full-rate only, 2x2 coarse only, and mixed (using a simple depth-variance heuristic).
   Step 4: Measure FPS, frame time, GPU bandwidth (via profiler), power consumption (10-minute sessions), and quality (PSNR, SSIM, LPIPS vs full-rate reference).
   Step 5: Analyze results to identify optimal tile size per architecture and scene type. Create a recommendation table for mobile developers.

6. Fallback Plan: If VRS tile size has minimal impact on the results, expand the analysis to include other VRS parameters such as the complexity scoring function, temporal smoothing factor, and minimum/maximum shading rate, turning it into a comprehensive mobile VRS parameter guide.

Example #2:
1. Title: Energy-Efficient Temporal Accumulation for Stable Frame Rate on Mobile Devices

2. Problem Statement: Mobile game rendering must maintain stable frame rates while minimizing power consumption to extend battery life. We propose a temporal accumulation framework that redistributes rendering work across frames based on content change detection.

3. Motivation: Current mobile rendering processes each frame independently, even when large portions of the screen have not changed between frames. By detecting which screen regions have changed and only re-rendering those regions while reusing previous frame data for static regions, we can reduce GPU work and power consumption without sacrificing perceived quality. This is inspired by display panel partial update technology but applied at the rendering level.

4. Proposed Method: After each frame, a lightweight compute shader compares the current and previous frame's color and depth buffers per-tile. Tiles with minimal change (below a perceptual threshold) are marked for reuse. In the next frame, only tiles marked as changed are rendered; reused tiles copy from the previous frame's color buffer. To handle gradual changes (e.g., time-of-day lighting), we force-refresh a small percentage of tiles each frame in round-robin order.

5. Step-by-Step Experiment Plan:
   Step 1: Implement per-tile change detection using compute shaders. Compare tiles using weighted color difference (CIE Delta E) and depth difference. Tile size: 32x32 pixels.
   Step 2: Implement selective tile rendering using Vulkan dynamic rendering with per-tile render area specification.
   Step 3: Test on 3 representative mobile game scenarios: (a) exploration game with slow camera movement, (b) UI-heavy menu screen, (c) fast-paced action gameplay.
   Step 4: Measure on Snapdragon 8 Gen 2 and Apple A17: FPS stability (variance), GPU utilization percentage, power consumption over 15-minute sessions, and perceived quality (no-reference metrics: NIQE, BRISQUE).
   Step 5: Compare against: (a) standard full-frame rendering, (b) simple frame skipping (render every other frame), (c) DRS (Dynamic Resolution Scaling).
   Step 6: Analyze the percentage of tiles reused across different game scenarios and the relationship between tile reuse rate and power savings.

6. Fallback Plan: If selective tile rendering introduces visible seams at tile boundaries or the overhead of change detection exceeds the savings, pivot to analyzing temporal coherence patterns in mobile games and publish a dataset of per-tile change statistics that could inform future mobile rendering optimizations.
